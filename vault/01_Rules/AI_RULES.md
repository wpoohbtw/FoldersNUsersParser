# AI Rules

## 1) Базовые принципы

1. Работаем строго в пределах задачи и затрагиваем только нужный скоуп.
2. Не добавляем "улучшения по пути" без отдельного запроса.
3. Не выполняем массовые рефакторинги в рамках точечной правки.
4. Все текстовые файлы проекта ведем в UTF-8.
5. Перед сдачей проверяем только те риски, которые относятся к текущей задаче.
6. Если задача маленькая, не запускать полный release/workflow-процесс.

---

## 2) Режимы задач

## Автоопределение режима задачи

Перед началом работы агент сам выбирает режим задачи.

Приоритет выбора:

1. Если пользователь явно указал режим — использовать его.
2. Если задача явно про релиз, деплой, полный аудит или финальную проверку — RELEASE TASK.
3. Если задача явно про новый дизайн, редизайн, визуальную концепцию, UI-kit, референсы или gpt-taste/lazyweb — DESIGN TASK.
4. Если задача требует изменения нескольких связанных частей проекта, логики, API или расследования бага — NORMAL TASK.
5. Во всех остальных случаях использовать QUICK FIX.

Если есть сомнение между QUICK FIX и NORMAL TASK — выбрать QUICK FIX.

Если есть сомнение между NORMAL TASK и DESIGN TASK — выбрать NORMAL TASK.

Если есть сомнение между NORMAL TASK и RELEASE TASK — выбрать NORMAL TASK.

Нельзя использовать lazyweb/gpt-taste только потому, что задача связана с UI. Для DESIGN TASK нужен явный запрос на дизайн, редизайн, визуальное направление или референсы.

Перед началом работы определить режим задачи.

### QUICK FIX

Использовать для мелких точечных задач:

* поправить текст;
* поправить CSS/отступ/цвет/размер;
* исправить мелкую ошибку в одном компоненте;
* изменить один элемент интерфейса;
* исправить локальный баг без изменения архитектуры;
* внести правку в 1–3 ожидаемых файла.

Правила QUICK FIX:

1. Не читать весь vault.
2. Не использовать lazyweb.
3. Не использовать gpt-taste.
4. Не читать `CHECKLIST_RELEASE.md`.
5. Не обновлять `CHANGELOG_AI.md`, если не менялась архитектура, API или терминология.
6. Не обновлять `API_MAP.md`, `GLOSSARY.md`, `COMPONENT_PATTERNS.md`.
7. Работать только с файлами, прямо связанными с задачей.
8. Сделать минимальный diff.
9. Перед сдачей проверить:

   * `git diff --name-only`;
   * что нет файлов вне скоупа;
   * если менялись русские тексты, проверить отсутствие моджибейки.

---

### NORMAL TASK

Использовать для обычных задач средней сложности:

* новая логика внутри существующего экрана;
* изменение поведения компонента;
* доработка существующего API;
* изменение нескольких связанных файлов;
* исправление бага, требующее анализа нескольких частей проекта.

Правила NORMAL TASK:

1. Читать только релевантные файлы проекта.
2. Читать `AI_RULES.md`.
3. Читать паттерны из `vault/02_Patterns` только если задача похожа на уже описанный паттерн.
4. Не использовать lazyweb и gpt-taste, если задача не связана с дизайном.
5. `CHECKLIST_RELEASE.md` читать только если пользователь просит финальную проверку, релиз или полный аудит.
6. Обновлять vault только если реально изменились:

   * архитектура;
   * API-контракты;
   * терминология;
   * устойчивый компонентный паттерн.
7. `CHANGELOG_AI.md` обновлять только для значимых изменений, а не для мелких фиксов.
8. Перед сдачей проверить diff и базовую работоспособность затронутого сценария.

---

### DESIGN TASK

Использовать только если задача явно связана с дизайном:

* новый экран;
* редизайн;
* визуальная система;
* layout;
* UI-kit;
* стилизация компонентов;
* улучшение визуального качества;
* подбор референсов.

Правила DESIGN TASK:

1. Можно использовать gpt-taste.
2. Можно использовать lazyweb, если нужны дизайн-референсы.
3. Сначала проверить существующий выбранный дизайн-направление проекта.
4. Не возвращаться к старым отвергнутым вариантам дизайна без запроса пользователя.
5. Если выбран один дизайн-вариант, считать его активным направлением.
6. Не менять backend/API без прямой необходимости.
7. Если появился устойчивый UI-паттерн, обновить `vault/02_Patterns/COMPONENT_PATTERNS.md`.
8. Если изменились дизайн-термины или названия сущностей, обновить `vault/03_Projects/GLOSSARY.md`.

---

### RELEASE TASK

Использовать только если пользователь явно просит:

* финальную проверку;
* подготовку к релизу;
* полный аудит;
* проверить весь проект;
* проверить перед деплоем.

Правила RELEASE TASK:

1. Читать `CHECKLIST_RELEASE.md`.
2. Выполнить полный scope-check.
3. Выполнить проверку на моджибейку.
4. Проверить критичные пользовательские сценарии.
5. Проверить, что vault обновлен по фактическим изменениям.
6. Обновить `CHANGELOG_AI.md`.
7. Если изменились API-контракты, обновить `API_MAP.md`.
8. Если изменилась терминология, обновить `GLOSSARY.md`.

---

## 3) Границы изменений

1. Если задача про frontend-экран, не менять backend-логику без прямой необходимости.
2. Если задача про backend/API, не менять UI-структуру без прямой необходимости.
3. Не менять соседние вкладки/страницы, если задача локальна.
4. Не выполнять массовый рефакторинг в рамках точечной правки.
5. Любое изменение вне исходного запроса сначала явно обозначить.
6. Если для исправления требуется выйти за скоуп, сначала объяснить причину.

---

## 4) Кодировка и анти-моджибейка

1. Любые пользовательские сообщения, UI-тексты, API errors и логи должны читаться без артефактов вида `РџС...`.
2. Проверку моджибейки выполнять обязательно, если:

   * менялись русские тексты;
   * менялись файлы локализации;
   * менялись API-ошибки;
   * менялись шаблоны сообщений;
   * пользователь прямо попросил проверку кодировки.
3. Для QUICK FIX без изменения текстов полную проверку кодировки не выполнять.
4. Команды проверки при необходимости:

```bash
rg -n "РџС|РќР|РЎР|СЃ|С‚|�" src backend
rg -n "charset|encoding|utf-8|utf8" src backend
```

5. Если найдена моджибейка:

   * исправить в рамках текущего скоупа, если это связано с задачей;
   * если не связано, отметить отдельно и не расширять diff без согласования.

---

## 5) Правила работы с Vault

Vault нужен для устойчивых решений, а не для каждой мелкой правки.

### Когда читать vault

1. Для QUICK FIX:

   * не читать весь vault;
   * читать только конкретный файл vault, если пользователь прямо на него сослался.

2. Для NORMAL TASK:

   * читать только релевантные файлы vault;
   * паттерны читать только при наличии похожего кейса.

3. Для DESIGN TASK:

   * читать активное дизайн-направление;
   * читать релевантные UI/component patterns.

4. Для RELEASE TASK:

   * читать `CHECKLIST_RELEASE.md`;
   * проверить актуальность проектных заметок.

### Когда обновлять vault

Обновлять vault только если появилось фактическое изменение:

1. Новый устойчивый компонентный паттерн → `vault/02_Patterns/COMPONENT_PATTERNS.md`.
2. Изменилась архитектура → проектные заметки в `vault/03_Projects`.
3. Изменился API-контракт → `vault/03_Projects/API_MAP.md`.
4. Изменилась терминология → `vault/03_Projects/GLOSSARY.md`.
5. Значимое изменение проекта → `vault/03_Projects/CHANGELOG_AI.md`.

Не обновлять vault для:

* мелких CSS-правок;
* исправления опечаток;
* изменения одного текста;
* локального визуального фикса;
* мелкого bugfix без изменения архитектуры/API/терминологии.

---

## 6) Использование lazyweb и gpt-taste

1. Не использовать lazyweb и gpt-taste автоматически.
2. Использовать gpt-taste только для DESIGN TASK.
3. Использовать lazyweb только если:

   * нужен новый дизайн;
   * нужен редизайн;
   * нужны внешние референсы;
   * пользователь прямо просит посмотреть референсы.
4. Не использовать lazyweb/gpt-taste для:

   * backend-задач;
   * API-задач;
   * мелких CSS-фиксов;
   * исправления багов;
   * изменения текста;
   * локальных правок существующего компонента.
5. Если дизайн-направление уже выбрано, не генерировать новые варианты без запроса пользователя.

---

## 7) Definition of Done

### QUICK FIX считается готовым, если:

1. Исправлена конкретная проблема.
2. Diff минимальный.
3. Нет изменений вне скоупа.
4. Если менялись русские тексты — нет признаков моджибейки.
5. В ответе указан список измененных файлов и кратко что проверено.

### NORMAL TASK считается готовым, если:

1. Реализован требуемый результат.
2. Нет лишних изменений вне скоупа.
3. Проверен основной затронутый сценарий.
4. Vault обновлен только если были реальные основания.
5. В ответе указано:

   * что изменено;
   * какие файлы затронуты;
   * что проверено.

### DESIGN TASK считается готовым, если:

1. Изменения соответствуют выбранному дизайн-направлению.
2. Нет случайной смены визуальной системы.
3. Не затронута backend/API-логика без необходимости.
4. Если появился устойчивый паттерн, он отражен в vault.
5. В ответе указано, какие визуальные решения применены.

### RELEASE TASK считается готовым, если:

1. Пройден `CHECKLIST_RELEASE.md`.
2. Проверены критичные сценарии.
3. Проверена кодировка.
4. Проверен diff.
5. Обновлены релевантные файлы vault.
